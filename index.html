Patrón de Código

Buenas Prácticas

1.

Las variables deben declararse en el menor escopo posible y más próximo a su utilización

  • Implementar las operaciones sobre una variable en un escopo pequeño, es más fácil controlar los efectos de la variable.

2.

El uso de números mágicos en el código debe ser evitado

  • Los números que no sean 0 ni 1 se declararán constantes. Si el número no tiene significado por sí mismo, utilizar una constante aumenta la legibilidad.
  • Otro enfoque es crear una función que devuelve el valor de la constante.

3.

Las expresiones condicionales complejas deben evitarse

  • Es mejor introducir variables booleanas temporales. De esta forma el programa gana una documentación automática.
  • El código es más fácil de leer, depurar y mantener.

Ejemplos:

bool isFinished = (elementNo < 0) || (elementNo > maxElement);
bool isRepeatedEntry = (elementNo == lastElement);
if (isFinished || isRepeatedEntry)
{
   ...
}

 }

 public void doSomethingElse(Object someParam) {
 // ...implementation goes here...
 }
}

1.

La indentación debe ser 4 espacios. No utilice el carácter TAB.

  • Los 'cases' dentro de un switch deben ser indentados.

2.

Las llaves que cierran clases, funciones y bloques tienen línea exclusiva.

  • No es obligatorio el uso de las llaves cuando hay una sola oración o en una función que cabe en una sola línea.

Ejemplos:

class Widget{
 public float getValue() {
  return value;
 }
}

3.

El número máximo de columnas por línea debe ser 100.

  • Si el contenido de la línea no cabe, pasar el resto a la línea inferior con una identación extra.
  • *No realizar el salto de línea cuando esto perjudica su legibilidad.

4.

Es recomendable que la declaración de los elementos de una clase tenga el siguiente orden: public, protected, private.

  • Tenga en cuenta que las palabras clave public, protected y private deben ser indentadas con la clase / struct.
  • Es preferible que las declaraciones de la sección se declaran en el orden siguiente: Tipos, métodos, atributos.

5.

Los operadores siempre deben estar entre dos espacios en blanco.

Ejemplos:

// Bad:
auto soma = a+b;

// Good:
auto soma = a + b;

6.

Todos los paréntesis, corchetes y "ángle brackets" no deben tener espacios extras.

  • Sólo debe tener espacio después de la coma entre los parámetros.

Ejemplos:

// Bad:
myFunction( a, v1 + v2 );

// Good:
myFunction(a, v1 + v2);

7.

No debe haber espacio entre paréntesis, declaraciones y llamadas de métodos.

Ejemplos:

// Bad:
if (condition) doStuff (param1, param2);

// Good:
if(condition) doStuff(param1, param2);

8.

Mantener el mismo estilo de comentario dentro de un archivo.

9.

Los comentarios deben ser incluidos en la línea anterior al comando respetando su identación en el código.

  • Esto evita que los comentarios rompen la estructura lógica del programa.
  • Excepción: comandos o declaraciones que tienen varias líneas.

Ejemplos:

// Bad:
int x = 0, y = 0, z = 0; // World Coordinates

// Good:
// Coordenadas de mundo
int x = 0, y = 0, z = 0;

// Except:
const unsigned short supportedProductIDList[] =
{
    0x0309,   // Cintiq Hybrid
    0x00f6,   // Cintiq 24HDT
    0x005e,   // Cintiq 22HDT
    0x032c    // Cintiq 27QHDT
}

Estilo: Nomenclatura

1.

Comentarios de clases y métodos en archivos de header deben seguir el patrón JavaDoc

Ejemplo:

 
 
 /**
  * ...method doSomethingElse documentation comment...
  * @param1 someParam description
  * @param2 someParam description
  */
 public void doSomethingElse(Object param1, Object param2) {
  // ...implementation goes here...
 }

2.

El código debe ser escrito en ingles y los comentarios en español.

3.

Las nomenclaturas de tipos, variables, valores de enums y métodos utilizan CamelCase.

  • Tipos (structs, classes, enums, etc.) deben ser escritos con CamelCase, comezando con mayúscula.
  • Variables y funciones/métodos deben ser escritas con camelCase, comenzando con minúscula.
  • Es preferible que funciones y métodos comiencen con verbos, pues representan ações.
